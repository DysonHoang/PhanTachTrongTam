02

Ý tưởng thuật toán

Ý tưởng cốt lõi: Thay vì thực hiện chặt nhị phân độc lập cho từng truy vấn, ta duy trì đồng thời khoảng tìm kiếm [L[i], R[i]] cho mọi truy vấn i. Trong mỗi vòng lặp, ta gom tất cả truy vấn có cùng mid lại và kiểm tra chúng trong một lượt duyệt.
1

Khởi tạo khoảng tìm kiếm

Với mỗi truy vấn i, đặt khoảng tìm kiếm ban đầu là L[i] = 0, R[i] = Q. Tất cả truy vấn đều bắt đầu với không gian tìm kiếm đầy đủ.

2

Gộp truy vấn theo mid

Tính mid = (L[i]+R[i])/2 cho từng truy vấn. Thêm truy vấn i vào danh sách check[mid] để xử lý chung.

3

Duyệt một lần — xử lý nhiều truy vấn

Duyệt tuần tự từ 1 đến Q. Tại bước q, thực hiện cập nhật vào CTDL, rồi với mọi truy vấn i ∈ check[q]: kiểm tra điều kiện và cập nhật L[i] hoặc R[i].

4

Lặp log₂Q lần → Hội tụ

Lặp lại bước 2-3 tổng cộng ≤ 20 lần (vì log₂(10⁵) ≈ 17). Khi L[i] > R[i], truy vấn i đã hội tụ về kết quả.

pseudo.cpp 
// Khung cài đặt tổng quát — Không đệ quy
for (int lap = 1; lap <= 20; lap++) {
    reset_ctdl();          // Reset cấu trúc dữ liệu
    
    // Gom truy vấn theo mid
    for (int i = 1; i <= Q; i++)
        if (L[i] <= R[i])
            check[(L[i]+R[i]) / 2].push_back(i);
    
    // Duyệt một lượt — xử lý tất cả
    for (int q = 1; q <= Q; q++) {
        update_ctdl(q);    // Thêm cập nhật thứ q
        for (int i : check[q]) {
            if (check_condition(i))
                R[i] = q - 1;  // Điều kiện thỏa → tìm sớm hơn
            else
                L[i] = q + 1;  // Chưa thỏa → tìm muộn hơn
        }
    }
}
03

Cài đặt chi tiết

⚠ Lưu ý quan trọng: Ở mỗi vòng lặp, ta phải reset lại toàn bộ cấu trúc dữ liệu (DSU, BIT, Segment Tree...) về trạng thái ban đầu, vì ta cần duyệt lại từ đầu cho chuẩn xác.

CTDL phổ biến

  • DSU — Union-Find cho bài liên thông
  • BIT/Fenwick — Đếm/tổng prefix
  • Segment Tree — Range update/query
  • DSU Small-to-Large — Gộp tập hợp

Mảng cần chuẩn bị

  • L[i], R[i] — Khoảng tìm kiếm truy vấn i
  • kq[i] — Kết quả của truy vấn i
  • check[q] — Danh sách truy vấn cần kiểm tra tại q
Độ phức tạp tổng quát: O( (N + Q) × log₂Q × f(N) )

01 VOSPLAY — Thời điểm liên thông → VNOI OJ

N học sinh, M game, Q phút — mỗi phút nối 2 máy. Tìm thời điểm sớm nhất mỗi game liên thông.

DSU Binary Search Parallel BS O((N+M+Q)·log Q)
Ý tưởng: Mỗi game ↔ một truy vấn binary search trên thời gian [0, Q]. Vì số cạnh tăng đơn điệu, sau thời điểm liên thông thì luôn liên thông → tính chất đơn điệu → áp dụng PBS. CTDL: DSU để kiểm tra liên thông.
vosplay.cpp 
for(int lap = 1; lap <= 20; lap++) {
    // Reset DSU
    for(int i = 1; i <= n; i++) lab[i] = -1;
    for(int i = 1; i <= q; i++) check[i].clear();
    
    // Gom truy vấn theo mid
    for(int i = 1; i <= m; i++)
        if(np[i].l <= np[i].r)
            check[(np[i].l + np[i].r) / 2].push_back(i);
    
    // Duyệt từng cạnh (cập nhật DSU)
    for(int i = 1; i <= q; i++) {
        Union(edge[i].F, edge[i].S);
        for(int j : check[i]) {
            bool ok = 1;
            int x = Find(like[j][0]);
            for(int k : like[j])
                if(Find(k) != x) ok = 0;
            if(ok) np[j].r = i-1, np[j].kq = i;
            else   np[j].l = i+1;
        }
    }
}
ĐPT: O((N + M + Q) · log₂Q)
02 MEDIAN — Trung vị đoạn con → VNOI OJ

Mảng N phần tử, Q truy vấn (l, r): tìm trung vị đoạn [l, r].

BIT/Fenwick Parallel BS O((N+Q)·log²N)
Ý tưởng: Trung vị là giá trị nhỏ nhất x sao cho số phần tử ≤ x trong [l,r] ≥ ⌊(r-l+2)/2⌋. Ta binary search trên giá trị x ∈ [1, 10⁵]. Dùng BIT: duyệt mảng theo thứ tự tăng dần, đánh dấu chỉ số → đếm số phần tử ≤ mid trong [l, r].
median.cpp (core) 
for(int lap = 1; lap <= 20; lap++) {
    fill(fen+1, fen+n+1, 0);
    d.clear();
    for(int i = 1; i <= q; i++)
        if(b[i].l <= b[i].r)
            d.push_back({(b[i].l+b[i].r)/2, i});
    sort(d.begin(), d.end()); // Sắp xếp theo mid
    
    int j = 1;
    for(auto [m, i] : d) {
        // Thêm phần tử có giá trị <= mid vào BIT
        while(j <= n && a[j].F <= m) upd(a[j].S), j++;
        
        int le=que[i].F, ri=que[i].S;
        int c = get(ri) - get(le-1); // Đếm số pt <= mid trong [l,r]
        
        if(c >= (ri-le+2)/2) b[i].kq = m, b[i].r = m-1;
        else b[i].l = m+1;
    }
}
03 MEDIAN2 — Tổng k phần tử nhỏ nhất

Mảng N phần tử, Q truy vấn (l, r, k): tính tổng k giá trị nhỏ nhất trong đoạn [l, r].

3× BIT Parallel BS O(20·(Q+N·log N))
Ý tưởng: Chặt nhị phân để tìm giá trị nhỏ nhất x sao cho có ≥ k phần tử ≤ x trong [l, r]. Dùng 3 BIT song song: (1) đếm số pt ≤ mid, (2) đếm số pt = mid, (3) tổng các pt < mid. Từ đó tính tổng k phần tử nhỏ nhất chính xác.
// Công thức tính tổng
Điều kiệnCông thức tổng
t = kans = t₂ + mid × t₁
t > k && t-t₁ ≤ kans = t₂ + mid × (k-(t-t₁))

t = #phần tử ≤ mid, t₁ = #phần tử = mid, t₂ = tổng pt < mid

Đồ thị N đỉnh M cạnh (có trọng số đông đúc). Q truy vấn (x, y, k): tìm đường x→y có ≥ k loại công trình, giá trị đông đúc nhỏ nhất.

DSU Small-to-Large unordered_set Parallel BS O((N+M+Q)·log²N)
Ý tưởng: Sắp xếp cạnh theo trọng số tăng dần. PBS trên không gian tìm kiếm là chỉ số cạnh đã sắp xếp. Với mỗi mid: thêm cạnh theo thứ tự, dùng DSU Small-to-Large lưu tập hợp loại công trình ở mỗi component. Kiểm tra x, y cùng component và component có ≥ k loại không.
🔑 Tại sao dùng unordered_set thay set? std::set có mọi thao tác O(log N) → độ phức tạp tăng lên O((N+M+Q)·log³N), có thể TLE. unordered_set gần O(1) mỗi thao tác, tổng thể O((N+M+Q)·log²N) vừa đủ.
05 SRAIN — Thùng nước mưa

N thùng nước có sức chứa giới hạn. Q sự kiện: mưa (tăng lượng nước đoạn [L, H] thêm d) hoặc truy vấn tổng lượng nước đoạn [L, H].

Segment Tree Lazy BIT Parallel BS O(Q·log²N)
Giai đoạn 1 — PBS: Với mỗi thùng i, dùng PBS tìm cơn mưa đầu tiên làm thùng đầy. BIT prefix sum mô phỏng lượng nước tích lũy.
Giai đoạn 2 — Mô phỏng: Segment Tree Lazy với mỗi nút lưu: val (tổng nước), lazy (lượng chờ cộng), cnt (số thùng chưa đầy). Khi thùng đầy: fix giá trị = b[i], cnt--. Update chỉ áp dụng cho thùng chưa đầy.
06

Kết luận & Tài liệu

✅ Khi nào dùng PBS?

  • Có Q truy vấn cùng dạng binary search
  • Điều kiện kiểm tra có tính đơn điệu
  • CTDL có thể được reset & xây lại hiệu quả
  • Q, N đủ lớn để O(Q·f) TLE nhưng O(f·logQ) AC

❌ Khi nào không dùng?

  • Các truy vấn phụ thuộc nhau (cần kết quả truy vấn trước)
  • CTDL không thể reset nhanh về trạng thái ban đầu
  • Q nhỏ → binary search thông thường đủ dùng
💡 Tóm gọn chiến lược: PBS = Binary Search × Q truy vấn, nhưng thực hiện song song trong log N vòng, mỗi vòng duyệt qua CTDL đúng một lần. Chi phí tăng thêm so với binary search đơn lẻ là đúng hệ số log N, nhưng ta xử lý được toàn bộ Q truy vấn.
